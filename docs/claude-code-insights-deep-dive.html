逆向分析 Claude Code CLI v2.1.69 中 /insights 命令的完整实现机制、数据流和所有内部 Prompt

CLI 版本: 2.1.69 二进制格式: Mach-O arm64 分析日期: 2026-03-06
概览 架构 数据流 数据 Schema 枚举映射 关键函数 实现细节 最终注入 附录: 完整 Prompt

01概览

/insights 是 Claude Code CLI 内置的分析命令，通过两阶段 LLM 管道生成个性化使用报告。

命令注册定义

{
  type:              "prompt",        // 不是 "local-jsx"，是 prompt 类型
  name:              "insights",
  description:       "Generate a report analyzing your Claude Code sessions",
  contentLength:     0,
  isEnabled:         () => true,
  isHidden:          false,
  progressMessage:   "analyzing your sessions",
  source:            "builtin",
  getPromptForCommand(T) { ... }     // 核心：预处理后生成注入 prompt
}
关键区分：/insights 的 type 是 "prompt"，而 /stats 的 type 是 "local-jsx"（直接渲染本地 UI 组件）。/insights 会预处理数据并生成 prompt，注入到当前对话中，由模型来呈现结果。

02架构概览

整体是一个"CLI 预处理 + LLM 分析 + 模型展示"的三层管道架构。

P1
Stage 1: 数据收集与提取

纯本地计算。扫描 JSONL 会话日志，提取结构化元数据（工具使用、代码变更、时间分布等），缓存到 session-meta/。

P2
Stage 2: LLM Facet 提取

对每个 session 的会话记录调用 Claude API，提取定性分析（目标、满意度、摩擦点等），缓存到 facets/。

P3
Stage 3: 聚合统计

纯本地计算。合并所有 session-meta + facets，计算总量、分布、排名等聚合指标。

P4
Stage 4: LLM 报告生成

将聚合数据分发给 7+1 个并行 LLM 调用，生成多维度叙事报告，渲染 HTML 并注入对话。

LLM 调用总量 = 未缓存的 session 数（facet 提取）+ 7 个维度报告 + 1 个 at_a_glance 总结 = N + 8 次 API 调用。所有调用均标记 querySource: "insights"。

03完整数据流

从原始会话日志到最终报告的 6 个处理阶段。

Phase 1 — 收集

扫描原始 Session 日志

函数 C14() 遍历 ~/.claude/projects/*/ 下所有目录，读取 .jsonl 会话文件。排除 agent-*.jsonl。按修改时间排序，最多处理最近 200 个未缓存 session。

↓
Phase 2 — 提取元数据

结构化指标提取 (UaR 函数)

对每个 session 提取：时间/时长、消息数、token 数、工具调用统计、语言检测、git 操作、代码变更行数、用户中断次数、响应时间分布。结果缓存到 session-meta/{uuid}.json。批量处理 50 个/批。

↓
Phase 3 — Facet 提取 (LLM)

定性分析提取 (Y14 函数)

对每个有意义的 session（messages ≥ 2 且 duration ≥ 1min），将完整会话记录发送给 Claude API，使用 Facet 提取 Prompt 提取：underlying_goal、goal_categories、outcome、satisfaction、friction、helpfulness。最多处理 50 个未缓存 session，10 个/批并行。结果缓存到 facets/{uuid}.json。

↓
Phase 4 — 聚合

统计聚合 (W14 函数)

纯本地计算。合并所有 session-meta + facets 数据，生成：总量统计、工具排行、语言分布、目标分类、结果分布、满意度分布、摩擦点统计、成功模式、并行使用检测（multi-clauding）、session 摘要列表（前50个）。过滤掉 warmup_minimal 类型的 session。

↓
Phase 5 — 报告生成 (LLM)

7+1 维度并行分析 (G14 → VK7)

将聚合数据打包为 JSON，分发给 7 个并行 LLM 调用（project_areas, interaction_style, what_works, friction_analysis, suggestions, on_the_horizon, fun_ending），每个 maxTokens=8192。完成后，将前几个维度的结果拼入 at_a_glance 的第 8 次 LLM 调用。

↓
Phase 6 — 渲染与注入

HTML 生成 + Prompt 注入

v14() 将所有分析结果渲染为自包含 HTML 报告（内嵌 CSS+JS），保存到 ~/.claude/usage-data/report.html。getPromptForCommand() 将完整 JSON 结果 + Markdown 摘要 + 固定输出指令拼装为 prompt，注入到当前对话上下文。

Multi-Clauding 检测 (j14)

检测用户是否在同一时间段使用多个 Claude Code 实例。算法：

  1. 收集所有 session 的 user_message_timestamps
  2. 按时间戳排序
  3. 维护 30 分钟（1800000ms）滑动窗口
  4. 检测窗口内不同 session 的消息交错出现
  5. 输出：重叠事件数、涉及的 session 数、重叠期间的消息数

Per-Session A1. Facet 提取 Prompt (A14)

▼ 
Analyze this Claude Code session and extract structured facets.

CRITICAL GUIDELINES:
1. **goal_categories**: Count ONLY what the USER explicitly asked for.
   - DO NOT count Claude's autonomous codebase exploration
   - DO NOT count work Claude decided to do on its own
   - ONLY count when user says "can you...", "please...", "I need...", "let's..."

2. **user_satisfaction_counts**: Base ONLY on explicit user signals.
   - "Yay!", "great!", "perfect!" → happy
   - "thanks", "looks good", "that works" → satisfied
   - "ok, now let's..." (continuing without complaint) → likely_satisfied
   - "that's not right", "try again" → dissatisfied
   - "this is broken", "I give up" → frustrated

3. **friction_counts**: Be specific about what went wrong.
   - misunderstood_request: Claude interpreted incorrectly
   - wrong_approach: Right goal, wrong solution method
   - buggy_code: Code didn't work correctly
   - user_rejected_action: User said no/stop to a tool call
   - excessive_changes: Over-engineered or changed too much

4. If very short or just warmup, use warmup_minimal for goal_category

SESSION:
{完整的会话 JSONL 记录}

输出格式：JSON 对象，字段包括 underlying_goal, goal_categories, outcome, user_satisfaction_counts, claude_helpfulness, session_type, friction_counts, friction_detail, primary_success, brief_summary。由 bK7() 函数校验。

▼ 
Summarize this portion of a Claude Code session transcript. Focus on:
1. What the user asked for
2. What Claude did (tools used, files modified)
3. Any friction or issues
4. The outcome

Keep it concise - 3-5 sentences. Preserve specific details like file names,
error messages, and user feedback.

TRANSCRIPT CHUNK:
{会话记录片段}

用途：当 session 过长超过 token 限制时，先分段摘要再做 facet 提取。

Dimension 5 B5. suggestions

▼

1. **MCP Servers**: Connect Claude to external tools, databases,
   and APIs via Model Context Protocol.
   - How to use: Run `claude mcp add <server-name> -- <command>`
   - Good for: database queries, Slack integration, GitHub issue
     lookup, connecting to internal APIs

2. **Custom Skills**: Reusable prompts you define as markdown files
   that run with a single /command.
   - How to use: Create `.claude/skills/commit/SKILL.md` with
     instructions. Then type `/commit` to run it.
   - Good for: repetitive workflows - /commit, /review, /test,
     /deploy, /pr, or complex multi-step workflows

3. **Hooks**: Shell commands that auto-run at specific lifecycle events.
   - How to use: Add to `.claude/settings.json` under "hooks" key.
   - Good for: auto-formatting code, running type checks,
     enforcing conventions

4. **Headless Mode**: Run Claude non-interactively from scripts and CI/CD.
   - How to use: `claude -p "fix lint errors" --allowedTools "Edit,Read,Bash"`
   - Good for: CI/CD integration, batch code fixes, automated reviews

5. **Task Agents**: Claude spawns focused sub-agents for complex
